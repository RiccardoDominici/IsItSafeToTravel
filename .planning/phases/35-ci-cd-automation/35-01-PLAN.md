# Phase 35 Plan 01: Staggered Advisory Fetching and Workflow Update

## Objective

Update the GitHub Actions data pipeline workflow to handle 34+ advisory sources with staggered batch fetching, per-source failure isolation, and a comprehensive fetch summary report. Ensure the workflow completes within free-tier time limits (30 minutes).

## Context

- @src/pipeline/fetchers/index.ts — Current fetcher orchestration (all sources in parallel)
- @src/pipeline/run.ts — Pipeline entry point
- @.github/workflows/data-pipeline.yml — Current daily pipeline workflow

## Tasks

### Task 1: Implement staggered batch fetching in pipeline
type="auto"

Refactor `fetchAllSources` in `src/pipeline/fetchers/index.ts` to fetch advisory sources in staggered batches with configurable delays between groups, preventing rate limiting and IP blocking when hitting 30+ government websites.

**Changes:**
1. Group advisory fetchers into batches (base sources first, then tier1, tier2a, tier2b, tier3a, tier3b with delays between groups)
2. Add a configurable delay between advisory batches (default 3 seconds)
3. Keep non-advisory sources (worldbank, gpi, inform, reliefweb, gdacs) fetching in parallel as before
4. Log batch progress and timing
5. Each individual source failure is isolated and logged

**Done criteria:**
- fetchAllSources fetches non-advisory sources in parallel, then advisory tiers in staggered batches
- Each batch has a configurable delay between it and the next
- Individual source failures don't block other sources
- Fetch summary log shows per-source success/failure with timing

### Task 2: Update GitHub Actions workflow with enhanced reporting
type="auto"

Update `.github/workflows/data-pipeline.yml` to increase timeout for expanded source set, add fetch summary output, and improve error reporting for the 34+ source pipeline.

**Changes:**
1. Increase timeout-minutes from 20 to 30 to accommodate staggered fetching
2. Add a step that generates a fetch summary (sources succeeded/failed, total time)
3. Add workflow annotation warnings for failed sources
4. Add the existing data history files to git add pattern

**Done criteria:**
- Workflow timeout is 30 minutes
- Fetch summary step reports per-source status
- Failed sources generate GitHub Actions warnings (not errors, since partial success is expected)
- Workflow continues on partial source failures
