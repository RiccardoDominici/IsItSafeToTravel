# Phase 6: SEO, Performance, and Launch - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the site discoverable by search engines, fast on mobile, and ready for public traffic. This includes: JSON-LD structured data on all pages, unique meta descriptions, XML sitemap with hreflang, Open Graph/Twitter card tags, mobile-responsive polish across breakpoints, and Lighthouse 90+ on mobile. Does NOT include new features, content changes, regional drill-down (v2), or analytics/monitoring setup.

</domain>

<decisions>
## Implementation Decisions

### Structured data (TECH-02)
- Country pages: WebPage + Place JSON-LD schemas — Place captures geographic/travel intent, WebPage for general SEO
- Homepage: WebSite schema with SearchAction — enables Google sitelinks search box
- Methodology/legal pages: WebPage schema only — simple informational content
- JSON-LD injected in `<head>` via Base.astro or per-page frontmatter, rendered as `<script type="application/ld+json">`
- Country page Place schema includes: name, description, geo coordinates (if available from country data), aggregateRating-style safety score presentation

### Meta tags & social sharing
- Open Graph tags on every page: og:title, og:description, og:image, og:url, og:type, og:locale
- Twitter card: summary_large_image on all pages — the map/score visuals are compelling at large size
- Unique meta descriptions per country page: template-driven from score data (matches Phase 4 auto-generated content pattern) — e.g., "Italy safety score: 7.2/10. Low crime risk, moderate health concerns. Updated daily from 3+ public sources."
- OG image: static fallback image per page type (homepage map screenshot, country pages generic branded card) — dynamic OG image generation exceeds budget constraint
- All meta tags added to Base.astro via optional props, with sensible defaults

### Sitemap & indexing
- Configure `@astrojs/sitemap` integration with i18n option for hreflang annotations in sitemap XML
- Sitemap already generated by the existing integration — needs i18n locale configuration added
- robots.txt: allow all, point to sitemap URL
- Canonical URLs already implemented in Base.astro — verify they resolve correctly for all page types

### Mobile polish (TECH-01)
- Mobile-first approach using existing Tailwind breakpoints (sm/md/lg/xl)
- Map on small screens: full-width with adjusted initial zoom level for mobile viewport
- Touch targets: minimum 44px for all interactive elements (WCAG 2.1 AA requirement from PROJECT.md)
- Search dropdown: full-width on mobile, positioned below header, adequate touch spacing between results
- Country page: stack layout on mobile (hero → breakdown → advisories → trend → sources), side-by-side where appropriate on desktop
- Header: hamburger menu or compact nav on mobile if not already handled
- Typography: ensure readable font sizes on mobile (minimum 16px body text to prevent iOS zoom)

### Performance optimization (TECH-04, Lighthouse 90+)
- Font loading: `font-display: swap` with `<link rel="preload">` for Inter and DM Sans variable fonts — prevent render-blocking
- Image optimization: use Astro `<Image>` component where applicable, serve WebP format
- JS bundle: audit Fuse.js bundle size, lazy-load Search component with `client:visible` instead of `client:load` if not already
- CSS: Tailwind purge is default behavior; Astro handles critical CSS inlining
- D3/topojson map: verify lazy loading — map JS should only load when homepage is visited, not on every page
- Minimize third-party scripts — site should have near-zero external dependencies
- Target metrics: LCP < 2.5s, FID < 100ms, CLS < 0.1

### Claude's Discretion
- Exact JSON-LD field values and schema.org property mapping
- OG image dimensions and design (within branded constraints)
- robots.txt exact content
- Specific Lighthouse audit fixes beyond the identified areas
- Whether to add preconnect/prefetch hints
- Exact mobile breakpoint adjustments per component

</decisions>

<specifics>
## Specific Ideas

- Unique meta descriptions must be genuinely unique per country — not just "Safety score for {country}" repeated 248 times with different names
- JSON-LD should make country pages eligible for rich results in Google (Place schema + safety-related properties)
- Mobile experience is critical for travelers who check safety info on the go — this isn't just a checkbox, it's a core use case
- Font loading must not cause visible layout shift (CLS penalty)
- The site already has good SSG foundations — Phase 6 is polish and verification, not a rewrite

</specifics>
