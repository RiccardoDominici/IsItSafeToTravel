    **Issue 2 - Global safety chart:**
    The current chart is a static build-time SVG with no interactivity. The data spans from 2012 to present (564 points) which makes the chart unreadable.

    Fix in all 3 global safety pages (`en/global-safety.astro`, `it/sicurezza-globale.astro`, `es/seguridad-global.astro`):

    1. Filter `globalHistory` to only show the last 90 days of data (recent, relevant dates). Add after `const globalHistory = loadGlobalHistory();`:
       ```
       const recentHistory = globalHistory.filter(d => {
         const date = new Date(d.date + 'T00:00:00Z');
         const ninetyDaysAgo = new Date();
         ninetyDaysAgo.setDate(ninetyDaysAgo.getDate() - 90);
         return date >= ninetyDaysAgo;
       });
       ```
    2. Replace all subsequent references to `globalHistory` with `recentHistory` in the chart computation (uniqueGlobalScores, showChart, dates, lineGen, lastPoint, etc.)
    3. Add a client-side `<script>` block at the bottom (before closing `</Base>`) that adds tooltip interactivity to the static SVG:
       - On mousemove over the SVG, find nearest data point and show a tooltip div (absolutely positioned) with date and score
       - On mouseleave, hide the tooltip
       - Create a tooltip div next to the SVG: `<div class="relative">` wrapper around the SVG, with a hidden tooltip `<div>` inside
       - Pass the chart data points via a `data-points` attribute on the SVG wrapper (JSON array of `{date, score, x, y}` computed at build time)
       - The script queries `.global-chart-wrapper`, parses data-points, and attaches mousemove/mouseleave/touchstart handlers
       - Format date using locale-appropriate format (use the `data-lang` attribute on the wrapper)
    4. Keep the "stable" and "accumulating" fallbacks for edge cases, but use `recentHistory` for their conditions too

    **Issue 5 - Comparison search input:**
    The search input in the compare pages is created via JS `document.createElement('input')`. The issue is likely that the `input` event doesn't fire properly on some mobile browsers when the input is dynamically created and the virtual keyboard doesn't open.

    Fix in all 3 compare page files (`en/compare.astro`, `it/confronta.astro`, `es/comparar.astro`) in the `<script>` block, in the `renderSelector()` function:

    1. Add `inputMode="search"` attribute to the input element: `input.inputMode = 'search';`
    2. Add `autocomplete="off"` and `autocapitalize="off"`: `input.autocomplete = 'off'; input.autocapitalize = 'off';`
    3. The root cause is likely that `renderSelector()` is called inside `render()` which rebuilds the entire selector section (innerHTML = ''), destroying and recreating the input each time. When a country is selected, the input loses focus. This is expected. But the REAL issue may be that on initial page load without selections, the input exists but tapping it on mobile doesn't open keyboard.
    4. Add `input.setAttribute('enterkeyhint', 'search');` to help mobile keyboards
    5. After appending the input to the DOM, if this is the initial render with no selections, call `setTimeout(() => input.focus(), 0)` ONLY if there's no touch event (don't auto-focus on mobile as it's annoying - check `'ontouchstart' in window`)
    6. More critically: ensure the input wrapper and input itself don't have any CSS that could block touch events. Add `touch-action: manipulation` to the input style: `input.style.touchAction = 'manipulation';`
    7. Add a `click` event listener on the input that calls `input.focus()` as a fallback for browsers where the default focus-on-tap doesn't work
  </action>
